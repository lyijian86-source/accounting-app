# 数据备份与恢复设计

日期：2026-04-28

## 背景

当前项目的账目、分类和标签都保存在浏览器本地 `localStorage` 中，适合离线使用，但不具备长期可恢复能力。现有导出功能只能导出 CSV 和展示型 JSON，不能把当前数据完整恢复回来。

## 目标

本轮目标是建立“本地备份 -> 本地导入 -> 覆盖或合并恢复”的闭环，保证用户在换手机、清缓存或误删后，仍能通过备份文件恢复数据。

## 范围

本轮只做以下内容：

1. 新增恢复型备份 JSON 导出。
2. 新增恢复型备份 JSON 导入。
3. 支持导入时选择覆盖或合并。
4. 导入前显示备份摘要和风险提示。
5. 导入后显示结果反馈并立即刷新当前页面数据。

本轮不做：

1. 云端同步。
2. 自动备份。
3. 账号系统。
4. 加密。
5. 历史多格式迁移。

## 备份格式

新增正式备份格式：

```json
{
  "version": 1,
  "exportedAt": "2026-04-28T12:00:00.000Z",
  "records": [],
  "categories": [],
  "tags": []
}
```

要求：

1. `records/categories/tags` 都保留内部原始结构。
2. `version` 用于后续格式演进。
3. `exportedAt` 用于导入前摘要展示。

现有展示型 JSON 继续保留，但不作为恢复入口。

## 导入模式

### 覆盖

覆盖当前本地：

- `records`
- `categories`
- `tags`

适用于换机、重装和完整恢复。

### 合并

保守合并，不覆盖现有本地数据：

- `records`：按 `id` 去重，同 `id` 跳过
- `categories`：先按 `id` 去重，再按 `name + type` 去重
- `tags`：先按 `id` 去重，再按 `name` 去重

## 导入校验

导入前必须校验：

1. 文件能否被解析为 JSON。
2. 顶层是否包含 `version`。
3. `records/categories/tags` 是否为数组。
4. 记录项是否包含恢复所需字段。

校验失败时：

1. 不写入任何数据。
2. 保持当前数据不变。
3. 显示明确错误原因。

## 交互流程

数据入口升级为“数据管理”弹窗，包含：

1. 导出备份
2. 导出 CSV
3. 导出展示 JSON
4. 导入备份

导入流程：

1. 用户选择 `.json` 文件
2. 系统解析并显示摘要
3. 用户选择“覆盖当前数据”或“合并到当前数据”
4. 系统执行导入
5. 显示结果反馈

如果用户选择覆盖，必须显示风险提示，建议先导出当前备份。
